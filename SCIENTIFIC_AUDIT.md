# Ophiuchus Scientific Audit

Date: 2026-06-30

## Current Trust Status

Ophiuchus is still a candidate-screening workbench, not a final phase-identification or quantitative refinement tool.

This audit adds the first reproducible validation path against user-exported VESTA/reference peak tables. The goal is to validate simulated peak positions under explicit conditions before trusting CIF-derived screening results.

Latest conclusion: the earlier 70 degree intensity problem was real. It came from Ophi's lightweight fallback CIF simulator, whose intensity model over-amplified high-angle peaks because it does not reproduce the atomic form-factor/profile physics used by VESTA or pymatgen. The main analysis and validation paths now prefer an external Anaconda `pymatgen` calculator when available, and only fall back to the lightweight simulator when pymatgen cannot be used.

2026-06-30 update: the remaining high-angle intensity disagreement in the external pymatgen path was traced mainly to missing Debye-Waller damping when comparing pymatgen stick intensities against the VESTA `MONI.int` profile. A uniform Debye-Waller B factor of 4.0 brings the 70.07 and 74.80 degree peaks close to the VESTA reference for the current ZrFe6Ge4 case. This is documented in `VESTA_VALIDATION.md`; it is a calibration setting for this case, not a universal physical constant.

## What Was Checked

### Materials Project Harvesting

The harvester now writes structures system-by-system instead of waiting for all chemical systems to finish. For each completed chemical system, Ophi updates the local SQLite library and refreshes the UI table.

Deduplication is performed by Materials Project `source_id`; the local structure hash is also stored for later duplicate audits.

Known remaining issue: the user's last visible long-running harvest did not leave MP CIFs in the default library, so the old process likely ended before persistence or wrote to a different context. The new flow is instrumented to expose per-system progress and partial success.

### XRD Simulation Configuration

A central `XRDConfig` now exists for validation:

- radiation source: `CuKalpha1`
- wavelength: `1.54056 Angstrom`
- public angle unit: `degree_2theta`
- default validation range: `10-90 degree`
- normalization: `max_100`

This separates validation settings from scattered UI/CLI defaults.

### VESTA / Reference Import

Ophi can now import manually exported VESTA/reference peak tables:

- CSV with `two_theta,intensity`
- CSV with `phase,two_theta,intensity_norm`
- VESTA-like text tables containing `2-Theta`, `I(f)`, and optional `(h k l)`

For files such as `all_peak_positions.csv`, a phase filter can select only `VESTA simulated`.

### Pattern Comparison

The comparator reports:

- matched peaks
- missing reference peaks
- extra calculated peaks
- median absolute delta 2theta
- max absolute delta 2theta
- strong peak coverage
- extra strong peaks

Peak position is treated as the primary correctness target. Intensity agreement is diagnostic because different tools may use different correction factors.

This remaining difference is model-dependent: VESTA `.int` gives a profile peak height after its own broadening/output process, while pymatgen returns normalized stick intensities. This confirms that the high-angle intensity error is mainly a missing high-angle damping/profile-consistency issue, not a peak-position, Bragg-law, or CIF composition error.

## Mismatch Cause Audit

1. Wavelength mismatch: now explicit in `XRDConfig`; validation default is Cu Kalpha1, 1.54056 Angstrom.
2. 2theta unit bug: public outputs and validation use degrees 2theta. Matched VESTA peaks within about 0.005 degree argue against a radian/theta confusion for those peaks.
3. Bragg formula bug: matched peak positions are close, so the basic Bragg position calculation appears plausible.
4. CIF parsing bug: still possible for complex CIFs. The fallback parser is simple and should not be considered fully CIF-compliant.
5. Symmetry handling bug: likely for fallback only. The external pymatgen path handles CIF symmetry through pymatgen's structure reader/calculator.
6. Space-group setting issue: still possible for non-standard CIFs.
7. Peak merging/tolerance issue: likely contributes to missing VESTA peaks and one remaining extra strong calculated peak near 45.00 degree.
8. Intensity model difference: partially isolated. The fallback intensity model is simplified and can badly over-amplify high-angle peaks. External pymatgen fixes the worst fallback artifact, but exact VESTA profile peak heights still require matching line-broadening/profile settings.
9. Kalpha doublet issue: validation default is Kalpha1 only; if VESTA export used a different line model, this must be recorded in the reference file metadata.
10. Experimental import issue: not used as the validation standard in this audit. Experimental XRD remains for screening, not calculator validation.

## Known Limitations

- The current default validation simulator prefers external Anaconda pymatgen when available; set `OPHI_DISABLE_EXTERNAL_PYMATGEN=1` only when intentionally testing fallback behavior.
- In the current `ophi` environment, creating Matplotlib figures and enabling in-process pymatgen XRD caused hard process exits during smoke tests. These are environment-level failures and are not hidden. External Anaconda pymatgen is used to avoid that crash path.
- The fallback simulator is not a trusted replacement for VESTA/pymatgen for complex structures and should not be used for confidence-critical CIF screening.
- Validation has been run against one user-relevant ZrFe6Ge4 reference so far; the prompt's target of several simple reference structures remains future work.

## How To Validate With A User VESTA Export

Preferred example using a complete VESTA `.int` pattern:

```powershell
python -m ophiuchus xrd-validate `
  --cif ".\data\library\structures\local\local_362b46a32755d09f.cif" `
  --reference "C:\path\to\MONI.int" `
  --out-dir ".\results\validation_reports" `
  --label "ZrFe6Ge4_vs_MONI_full"
```

Selected peak CSV example:

```powershell
python -m ophiuchus xrd-validate `
  --cif ".\data\library\structures\local\local_362b46a32755d09f.cif" `
  --reference "C:\path\to\all_peak_positions.csv" `
  --phase "VESTA simulated" `
  --out-dir ".\results\validation_reports" `
  --label "ZrFe6Ge4_vs_VESTA"
```

If the reference file is only two columns, use:

```powershell
python -m ophiuchus xrd-validate `
  --cif ".\path\to\structure.cif" `
  --reference ".\path\to\simulated_peak_positions.csv" `
  --out-dir ".\results\validation_reports"
```
